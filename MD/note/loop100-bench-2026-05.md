## 2. 關鍵觀察

### 2.1 json-llvm 一致性高，跨平台 ~3-4 MIPS

GBA arm + thumb + GB json-llvm 三者都落在 **2.7-3.7 MIPS** 範圍。
表示 dispatch overhead 是主要瓶頸 — 不論 CPU 多複雜，每條指令
indirect-call 進 LLVM JIT'd code 的 cost 都差不多。

### 2.2 GB legacy 38 MIPS = json-llvm 14×

Hand-written switch-case backend的優勢：
- CPU branch predictor 友善（switch table 跳越早期 branch）
- 沒 indirect call overhead
- inline 邏輯，減少 register save/restore
- DMG instruction 簡單（沒 conditional execution / shifted operand）

### 2.3 cycle/instr 平均

- **GBA**：84.3M cycles ÷ 84.3M instr = **4 cycles/instr**
  （我們目前 hardcoded `cyclesPerInstr=4` per Step；對 ARM7TDMI 大多
  指令 1S=4t-cyc 是正確的近似，但對 LDM/STM 等多 cycle 指令低估）
- **GB legacy**：84.3M t-cycles ÷ 9.7M instr = **8.68 t-cyc/instr**
  （DMG 真實情況：簡單 LD = 4t-cyc，記憶體 = 8-12，PUSH/POP/CALL
  = 12-24，平均符合 ~8）
- **GB json-llvm**：84.3M t-cyc ÷ 9.8M instr = **8.60 t-cyc/instr**
  （差異在 HALT 處理：legacy/json-llvm 對 HALT 的 cycle accounting
  略不同，但都接近）

### 2.4 GBA 0.9× real-time 是 dispatch-bound

雖然 0.9× < 1.0× 看起來臨界點，但要注意：
- Loop100 跑 100 次 ARM/Thumb 全測 — 比真實遊戲負載高
- 真實遊戲多半 I/O wait + HALT（CPU 60% idle），實際遊戲更接近
  1.5-2× real-time
- BIOS LLE 開機已驗 1.0× real-time（之前 commit `33e1fa1`）

## 3. 跟 2026-05-01 baseline 比

對照 [`MD/note/performance-baseline-2026-05.md`](/MD/note/performance-baseline-2026-05.md) (Phase 4.5 完工時)：

| Backend | Phase 4.5 baseline | Phase 5.7 (this) | 變化 |
|---|---|---|---|
| GB legacy   | ~56 MIPS  | 38.4 MIPS | -32% |
| GB json-llvm| ~3.4 MIPS | 2.73 MIPS | -20% |
| GBA json-llvm| ~4.4 MIPS | 3.7 MIPS  | -16% |

Phase 5 加了：
- **scheduler.Tick** per instruction（GBA/GB 都新增） — 每 step 多
  一個 method call + scanline arithmetic
- **PPU LY/STAT/IRQ** 維護
- **OBJ window mask** + **per-layer composite + BLDCNT** 在 PPU 端
  （但只在 RenderFrame 才跑，每 frame 一次，影響小）
- **BIOS open-bus protection** + **NotifyExecutingPc** + 
  **NotifyInstructionFetch** hook（每 step 兩個 virtual-method 呼叫）
- **PcWritten flag** 處理 + branch 偵測強化
- **5 個 ARM bug fix**（每個多幾條 IR 指令）

合計 ~15-30% per-step overhead 增加，跟測量出來的 ~20-30% 衰退吻合。
這是「正確性換速度」的合理 trade-off。

## 4. Phase 7 block-JIT 預期改進

如果 block-JIT 把每 16-32 條指令連續編成一個 LLVM function，
dispatch overhead 攤掉：

| Backend | Current MIPS | Block-JIT 預期 | 加速 |
|---|---|---|---|
| GBA json-llvm  | 3.7  | ~30-50 | 8-13× |
| GB json-llvm   | 2.7  | ~25-40 | 9-15× |

依據：
- LLVM optimizer 在大 block 內可以做 register caching、CSE、
  constant folding、unreachable elimination
- 每 block dispatch 只 1 次（vs 每 instr 1 次） — 假設 block 平均
  20 instr，dispatch overhead 降 95%
- 有機會接近或超過 GB legacy 的 38 MIPS（因為 LLVM 的 codegen
  比 hand-written switch 更激進）

不過 Phase 7 實作有複雜度（block detector、SMC handling、
indirect branch、code cache），是 1-2 週工作量。

## 5. 實際應用面意義

對「test ROM CPU 驗證 + 截圖」這個 MVP 目標：

- ✅ **GBA arm-loop100 / thumb-loop100**：~23 s host time 跑完
  100 iterations，可接受
- ✅ **GB 09-loop100 (legacy)**：6 s host time 跑完 100 iterations，
  可接受
- ⚠ **GB 09-loop100 (json-llvm)**：~50 s host time 跑完 100 iterations，
  慢但可用 — 主要驗證 framework 通用性，不是日常開發

對「實時跑商業遊戲」：
- ❌ 目前 GBA json-llvm < 1× real-time → 跑商業遊戲會掉 frame
- ❌ Phase 7 block-JIT 才是解
- 不過這從 Phase 5 起就明確 cut 出 MVP 範圍

## 6. Bench 重現方法

```bash
# 1. Build all loop100 ROMs
tools/build_loop100.sh

# 2. Run each (1200 frames each)
dotnet run --project src/AprGba.Cli -c Release -- \
    --rom=test-roms/gba-tests/arm/arm-loop100.gba --frames=1200

dotnet run --project src/AprGba.Cli -c Release -- \
    --rom=test-roms/gba-tests/thumb/thumb-loop100.gba --frames=1200

dotnet run --project src/AprGb.Cli -c Release -- \
    --cpu=legacy \
    --rom=test-roms/gb-test-roms-master/cpu_instrs/loop100/09-loop100.gb \
    --frames=1200

dotnet run --project src/AprGb.Cli -c Release -- \
    --cpu=json-llvm \
    --rom=test-roms/gb-test-roms-master/cpu_instrs/loop100/09-loop100.gb \
    --frames=1200
```

每個輸出末尾印 `instructions: N → M.MM MIPS`。
