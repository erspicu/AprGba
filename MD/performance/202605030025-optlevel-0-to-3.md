# OptLevel 0 → 3（Phase 7 B.a）— **perf-neutral, kept**

> **策略**：把 MCJIT 的 OptLevel 從 0 (None) 直接拉到 3 (Aggressive)。
> 一行 config 改動，純加 LLVM optimisation 的成本，看 LLVM 對我們
> per-instruction function 能擠出多少效能。
>
> **Hypothesis**：O0 沒做任何優化，O3 跑全套 LLVM passes，per-instruction
> JIT'd function 應該縮小、執行加速。
>
> **結果**：**全部 4 個 ROM 的差異都在 ±3% 之內，明顯 measurement noise**。
> Hypothesis 不成立 — 原因是 per-instruction function 太小（每條指令一個
> function、每個 function 5–30 行 IR），LLVM optimizer 沒空間發揮。
> bottleneck 在 dispatcher overhead 不在 JIT 過的 native code 裡。
>
> **決定**：保留 O3（沒變慢，未來如果做 block-level JIT 才會真的有
> 用），記錄結論作為「OptLevel 不是當前瓶頸」的證據。

---

## 1. 結果（3-run 平均）

| ROM                     | Backend     | O0 baseline | **O3 (avg)** | runs | Δ      |
|-------------------------|-------------|------------:|-------------:|------|-------:|
| GBA arm-loop100.gba     | json-llvm   | 3.82        | **3.71** (3.68–3.77) | 3    | **−2.9%** |
| GBA thumb-loop100.gba   | json-llvm   | 3.75        | **3.73** (3.64–3.90) | 3    | **−0.5%** |
| GB 09-loop100.gb        | legacy      | 32.76       | **32.50** (31.85–33.13) | 3 | **−0.8%** |
| GB 09-loop100.gb        | json-llvm   | 2.66        | **2.70** (2.66–2.73) | 3    | **+1.5%** |

**Verdict**：**perf-neutral**。GB legacy 完全不走 LLVM 路徑，差異是純 noise；
其他三個 json-llvm path 也都 ±3% 內。所有 run 範圍跟 baseline 範圍重疊，
沒有統計顯著差異。

---

## 2. 為什麼沒效

把這件事當「實驗 disproof」對 Phase 7 後續策略很重要：

### 2.1 per-instruction function 對 LLVM optimizer 太小

我們目前架構：每條 spec 指令獨立一個 LLVM function。典型一個 function 含
3–8 條 IR instructions（read_reg → ALU → write_reg + 幾條 flag 更新）。
LLVM 主要 optimisation passes 的價值來自：

- **inlining**：把多 function 內聯展開 → 我們每 function 都已經是 leaf，
  沒有可內聯的 callees
- **CSE / GVN**：消除重複表達式 → 一個 function 內表達式太少，幾乎沒有
  redundancy 可消
- **loop optimisation**：unrolling / hoisting → 我們 function body 沒有 loop
- **scalar replacement of aggregates**：split struct → 我們 input/output
  都是 explicit GEP 進 CpuState struct，已經是 split 過的
- **dead code elimination**：移除 unused → spec 寫的 step 都有用
- **instruction combining**：合併 add/and 等 → 有些空間但很有限

LLVM 真正能做事的場景需要 **大 function** （20+ IR instructions）。
per-instruction granularity 把 optimizer 的肌肉廢掉。

### 2.2 Dispatcher overhead 才是當前瓶頸

每條指令的成本拆解（粗估，沒實測 profile）：
- Decode + dispatch: ~30-40 ns（hash lookup + indirect call）
- JIT'd function body: ~5-15 ns（小、簡單）
- Tick + IRQ check + PcWritten check: ~10-20 ns
- Total: ~50-80 ns/instr → 12-20 MIPS theoretical

我們實測 GBA json-llvm 3.7 MIPS = 270 ns/instr — 跟上面 estimate 差約
3-5×，差距吃在 .NET interop call、struct field GEP、memory bus extern
trampoline 等地方。

**OptLevel 解的是「JIT'd function body 那 5-15 ns」這塊**，但即使把它
壓到 0（不可能）總時間也只省 5-20%。要顯著加速必須打 dispatcher 那
30-40 ns + Tick 10-20 ns。

### 2.3 對 Phase 7 後續排序的暗示

| 策略 | 影響的 cost 區段 | 預期 ROI |
|---|---|---|
| **A. Block-level JIT** | dispatcher overhead 攤掉 | **高** — 直接打到瓶頸 |
| **B.b/c GEP CSE / micro-op fusion** | function body 微小化 | 低 — body 已小 |
| **B.d nuw/nsw hint** | function body | 低 |
| **B.e Tick / hook inlining** | 每條指令的固定 overhead | **中** — 直接打到 IRQ check |
| **C. Lazy flag** | flag 寫入 cost | **中** — 看 spec 寫多少 flag op |
| **D. Tier compilation** | 編譯時間 / hot block 重編 | 在 block-JIT 之後才有意義 |
| **E. Mem-bus fast path** | extern call cost | **中-高** — 每 mem op 省 trampoline |
| **F. Direct dispatcher table** | dispatcher overhead | **中** — 跟 A 部分重疊 |
| **G. Native AOT** | host runtime overhead | 中 — 對 GB legacy 那種 ~300ms run 影響大 |

排序：**A（block-JIT）→ E（mem-bus fast path）→ B.e（Tick inlining）
→ C（lazy flag）→ F（dispatcher table）→ G（AOT）→ B.b/c/d（IR-level
微調）**。先做 A 因為它一次性把 dispatcher overhead 整個 amortize，
其他大多數優化都會在 block-JIT 之後變得更顯著。

## 3. 為什麼還是保留 O3

雖然當下沒收益，但：

- **沒退步**：所有 4 個 ROM 在 noise 範圍內
- **未來的籌碼**：block-level JIT 一上來，function 變大（多條指令串接），
  O3 會立刻有意義 — 不必等到那時才再切回去
- **編譯時間**：實測 spec → IR → MCJIT 的 startup cost 從 ~200 ms (O0)
  升到 ~250-300 ms (O3)，可接受
- **單元測試 + Blargg 全綠**：345/345 unit + 9/9 Blargg cpu_instrs subtests
  在 json-llvm path 都過
