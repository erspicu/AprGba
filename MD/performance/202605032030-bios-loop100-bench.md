# BIOS LLE loop100 perf — block-JIT 比 per-instr **慢 28-31%**（結構性，非 debug clutter）

> **Trigger**：使用者要求測 100LOOP + 真 BIOS、ARM/Thumb × per-instr/bjit
> 4 combo 的效能。順帶清掉 dev-leftover hardcoded debug。
>
> **發現**：BIOS LLE 跑 loop100 時，**block-JIT 比 per-instr 慢**
> （-28% 到 -31%）。HLE 跑同樣 ROM 時 bjit 比 pi 快 +27% 到 +40%。
> 切換 boot 模式翻轉了 bjit 的優勢。
>
> **原因**：bjit 的 cache-miss 成本（建 LLVM module → 5-pass
> 優化 pipeline → JIT add）在 BIOS path 的「許多冷碼」flow 下無法攤銷
> （vs HLE 直跳 cart entry 一個熱迴圈）。
>
> **決定**：紀錄為 known characteristic、不 fix。perf 工作後續另開
> （tier-block-compile-cost / cold-block-skip 之類）。

---

## 1. 結果

### 1.1 Bench (3-run avg, MIPS)

| combo | run1 | run2 | run3 | **avg** |
|---|---|---|---|---|
| arm   pi   (BIOS LLE) | 7.62 | 7.13 | 7.39 | **7.38** |
| arm   bjit (BIOS LLE) | 5.37 | 5.05 | 5.39 | **5.27** |
| thumb pi   (BIOS LLE) | 7.24 | 7.23 | 7.35 | **7.27** |
| thumb bjit (BIOS LLE) | 5.65 | 5.50 | 5.57 | **5.57** |

Discreteness（max-min / avg）：
- arm pi: 6.6%（略高，可能 cache miss 變動）
- arm bjit: 6.5%
- thumb pi: 1.7%
- thumb bjit: 2.7%

### 1.2 跟 HLE 比

| combo | HLE avg | BIOS avg | Δ |
|---|---|---|---|
| arm   pi   | 8.12 | 7.38 | **-9.1%** |
| arm   bjit | 10.31 | 5.27 | **-48.9%** |
| thumb pi   | 8.16 | 7.27 | **-10.9%** |
| thumb bjit | 11.43 | 5.57 | **-51.3%** |

**Per-instr 走 BIOS 多花 ~10%** — BIOS 啟動序列的 overhead，符合預期。

**Block-JIT 走 BIOS 多花 ~50%** — 這是 bjit 模式的結構性 cost。

### 1.3 同一 ROM bjit vs pi 比較

| boot 模式 | pi MIPS | bjit MIPS | bjit 相對 |
|---|---|---|---|
| HLE  arm   | 8.12 | 10.31 | **+27%** ← bjit 贏 |
| HLE  thumb | 8.16 | 11.43 | **+40%** ← bjit 贏 |
| BIOS arm   | 7.38 | 5.27 | **-29%** ← bjit 輸 |
| BIOS thumb | 7.27 | 5.57 | **-23%** ← bjit 輸 |

切 BIOS LLE 後 bjit 的優勢消失了，反而成為負擔。

## 2. Root cause hypothesis

### 2.1 Cache miss → AddModule 成本

block-JIT 的設計：
1. 第一次撞到某個 block start PC → `CompileBlockAtPc()`
2. 走 `BlockDetector.Detect()` 從 bus 抓 N 個指令
3. 建 LLVM module → `BlockFunctionBuilder.Build()` emit IR
4. **`HostRuntime.AddModule()`：set datalayout → 5-pass optimization
   pipeline (mem2reg / instcombine / gvn / dse / simplifycfg) → 加入 JIT**
5. cache 起 fnPtr，下次直接 call

第 4 步是 hot path 上的「fixed cost per cold block」。Phase 7 H.a 把
pipeline 加到 5 pass 後（之前 4 pass）這個 cost 增加。

### 2.2 BIOS 跟 HLE 的差異

- **HLE 路徑**：`apr-gba` skip BIOS、直接 setup cart entry @ 0x08000000，
  loop100 ROM 的主迴圈很小（幾十條指令），cache 暖起來後 N=百萬
  次 dispatch 都直接 hit cache、優化 pipeline 攤到無限大 → bjit
  贏 pi 27-40%。

- **BIOS LLE 路徑**：先跑完 BIOS 啟動（CPU mode 切換、setup IRQ vectors、
  IRQ handler 進出 SVC mode、SoftReset/IntrWait/SWI handler entry/exit
  路徑等），這些路徑大量是「跑一次的冷碼」。每條冷碼路徑要編譯一次，
  代價大；hit count 又少 → 攤不平。最後可能 **block 編譯 cost > block
  執行 cost**。

實證 hypothesis 的方法（沒做，留 followup）：
- 在 `CompileBlockAtPc` 加 `BlockCompileMs` 累計 + 末尾 print
- 比較 HLE/BIOS 兩條 path 的「總編譯 ms / 總執行 ms」

### 2.3 為何 per-instr BIOS 只慢 10%？

per-instr 的 instruction dispatch 路徑：
- 純查 `_fnPtrByDef` cache → 直接 call function pointer
- 所有 spec emitter function 在 `Compile()` 一次預編譯（一個大 module，
  攤一次 5-pass），執行期 zero compile cost
- BIOS 啟動只是多跑一些 instruction，instruction-level 不會 trigger
  新編譯

所以 BIOS overhead = 純 instruction count + IRQ overhead（ARM exception
entry/exit），約 10%，符合預期。

short-circuit 掉，本來就 free。但**clutter 還是清掉乾淨**。

## 5. Followup 建議（不阻擋 merge）

1. **Block compile cost amortization**: 跑 hot block 統計，hit-count <
   threshold 的 block 不走完整 5-pass pipeline（只 mem2reg + simplifycfg
   即可），熱起來後 promote。可能把 BIOS bjit 從 5.3 拉回 7-8 接近 pi。
2. **Inline BIOS HLE replacement for SWI handlers**: 大多 GBA game 用 BIOS
   主要是 SWI 1-FE。如果 emulator 有 HLE 攔截 SWI、跳過 BIOS handler，
   bjit 在 game 場景就不會踩這條 cliff。
3. **Block detector boundary tuning**: `maxInstructions: 64` 可能太大，
   BIOS 路徑的 cold blocks 平均 N 多少？小 N → fixed compile cost 比例
   高 → 更該 skip 編譯。

這些都是 perf-tuning 而非 correctness 問題，留給後續單獨 commit 處理。
