## 2. Key observations

### 2.1 json-llvm consistency is high, ~3-4 MIPS across platforms

GBA arm + thumb + GB json-llvm all land in the **2.7-3.7 MIPS** range.
This means dispatch overhead is the main bottleneck — regardless of
how complex the CPU is, the cost of each indirect call into LLVM
JIT'd code is similar.

### 2.2 GB legacy 38 MIPS = 14x json-llvm

Hand-written switch-case backend's advantages:
- CPU branch predictor friendly (switch table jumps via early branches)
- No indirect call overhead
- Inline logic, fewer register save/restores
- DMG instructions are simple (no conditional execution / shifted operand)

### 2.3 Cycle/instr averages

- **GBA**: 84.3M cycles / 84.3M instr = **4 cycles/instr**
  (we currently hardcode `cyclesPerInstr=4` per Step; for ARM7TDMI
  most instructions 1S=4t-cyc is a correct approximation, but
  underestimates multi-cycle instructions like LDM/STM)
- **GB legacy**: 84.3M t-cycles / 9.7M instr = **8.68 t-cyc/instr**
  (DMG actuality: simple LD = 4t-cyc, memory = 8-12, PUSH/POP/CALL
  = 12-24, average matches ~8)
- **GB json-llvm**: 84.3M t-cyc / 9.8M instr = **8.60 t-cyc/instr**
  (the difference is in HALT handling: legacy/json-llvm differ
  slightly in HALT cycle accounting, but both are close)

### 2.4 GBA 0.9x real-time is dispatch-bound

Although 0.9x < 1.0x looks borderline, note:
- loop100 runs 100 full ARM/Thumb iterations — higher than real game load
- Real games are mostly I/O wait + HALT (CPU 60% idle); actual games
  would land closer to 1.5-2x real-time
- BIOS LLE boot already verified at 1.0x real-time (earlier commit `33e1fa1`)

## 3. Comparison vs 2026-05-01 baseline

Compared to [`MD_EN/note/performance-baseline-2026-05.md`](/MD_EN/note/performance-baseline-2026-05.md) (Phase 4.5 wrap-up):

| Backend | Phase 4.5 baseline | Phase 5.7 (this) | Change |
|---|---|---|---|
| GB legacy   | ~56 MIPS  | 38.4 MIPS | -32% |
| GB json-llvm| ~3.4 MIPS | 2.73 MIPS | -20% |
| GBA json-llvm| ~4.4 MIPS | 3.7 MIPS  | -16% |

Phase 5 added:
- **scheduler.Tick** per instruction (added on both GBA/GB) — each
  step gets one extra method call + scanline arithmetic
- **PPU LY/STAT/IRQ** maintenance
- **OBJ window mask** + **per-layer composite + BLDCNT** in PPU side
  (but only runs in RenderFrame, once per frame, low impact)
- **BIOS open-bus protection** + **NotifyExecutingPc** +
  **NotifyInstructionFetch** hooks (two virtual-method calls per step)
- **PcWritten flag** handling + branch detection hardening
- **5 ARM bug fixes** (each adds a few IR instructions)

Sums to ~15-30% per-step overhead increase, matching the measured
~20-30% regression. This is a reasonable "correctness for speed"
trade-off.

## 4. Phase 7 block-JIT expected improvements

If block-JIT compiles every 16-32 consecutive instructions into a
single LLVM function, dispatch overhead is amortised:

| Backend | Current MIPS | Block-JIT projected | Speedup |
|---|---|---|---|
| GBA json-llvm  | 3.7  | ~30-50 | 8-13x |
| GB json-llvm   | 2.7  | ~25-40 | 9-15x |

Reasoning:
- LLVM optimiser can do register caching, CSE, constant folding,
  unreachable elimination inside large blocks
- Each block dispatches only once (vs once per instr) — assuming
  blocks average 20 instructions, dispatch overhead drops 95%
- Possible to approach or exceed GB legacy's 38 MIPS (because LLVM's
  codegen is more aggressive than hand-written switch)

That said, Phase 7 implementation has complexity (block detector,
SMC handling, indirect branch, code cache) — 1-2 weeks of work.

## 5. Practical implications

For the "test ROM CPU verification + screenshot" MVP goal:

- **GBA arm-loop100 / thumb-loop100**: ~23s host time to run 100
  iterations, acceptable
- **GB 09-loop100 (legacy)**: 6s host time to run 100 iterations,
  acceptable
- **GB 09-loop100 (json-llvm)**: ~50s host time to run 100 iterations,
  slow but usable — primarily verifies framework generality, not
  daily development

For "running commercial games in real-time":
- Currently GBA json-llvm < 1x real-time -> commercial games will drop frames
- Phase 7 block-JIT is the answer
- This was explicitly cut from MVP scope starting Phase 5

    --rom=test-roms/gba-tests/arm/arm-loop100.gba --frames=1200

    --rom=test-roms/gba-tests/thumb/thumb-loop100.gba --frames=1200

    --cpu=legacy \
    --rom=test-roms/gb-test-roms-master/cpu_instrs/loop100/09-loop100.gb \
    --frames=1200

    --cpu=json-llvm \
    --rom=test-roms/gb-test-roms-master/cpu_instrs/loop100/09-loop100.gb \
    --frames=1200
