# OptLevel 0 → 3 (Phase 7 B.a) — **perf-neutral, kept**

> **Strategy**: bump MCJIT's OptLevel from 0 (None) directly to 3 (Aggressive).
> One-line config change, pure cost of running the LLVM optimisation, see
> how much LLVM can squeeze from our per-instruction functions.
>
> **Hypothesis**: O0 does no optimisation, O3 runs the full LLVM pass
> pipeline; per-instruction JIT'd functions should shrink and execute faster.
>
> **Result**: **All 4 ROMs differ within ±3%, clearly measurement noise.**
> Hypothesis doesn't hold — reason is the per-instruction function is too
> small (one function per instruction, each function 5–30 lines IR), the
> LLVM optimizer has no room to work. Bottleneck is in dispatcher overhead,
> not in JIT'd native code.
>
> **Decision**: keep O3 (didn't get slower, will be useful when block-level
> JIT lands), record the conclusion as evidence that "OptLevel is not the
> current bottleneck".

## 2. Why it didn't work

Treating this as an "experiment disproof" matters for ordering Phase 7
strategies:

### 2.1 Per-instruction function is too small for the LLVM optimizer

Our current architecture: each spec instruction is an independent LLVM
function. A typical function contains 3–8 IR instructions
(read_reg → ALU → write_reg + a few flag updates).
Value of the main LLVM optimisation passes:

- **inlining**: expand multiple functions inline → all of ours are leaf
  functions, no callees to inline
- **CSE / GVN**: eliminate redundant expressions → too few expressions
  in one function, almost no redundancy
- **loop optimisation**: unrolling / hoisting → no loops in our function bodies
- **scalar replacement of aggregates**: split struct → our input/output is
  already explicit GEP into the CpuState struct, already split
- **dead code elimination**: remove unused → spec-written steps are all used
- **instruction combining**: merge add/and etc. → some room but very limited

LLVM does its real work on **large functions** (20+ IR instructions).
Per-instruction granularity neuters the optimizer's muscle.

### 2.2 Dispatcher overhead is the current bottleneck

Cost breakdown per instruction (rough estimate, no actual profile):
- Decode + dispatch: ~30–40 ns (hash lookup + indirect call)
- JIT'd function body: ~5–15 ns (small, simple)
- Tick + IRQ check + PcWritten check: ~10–20 ns
- Total: ~50–80 ns/instr → 12–20 MIPS theoretical

Measured GBA json-llvm 3.7 MIPS = 270 ns/instr — about 3–5× the estimate,
the gap eaten by .NET interop call, struct field GEP, memory bus extern
trampoline, etc.

**OptLevel addresses the "JIT'd function body 5–15 ns" portion**, but
even compressing it to 0 (impossible) would only save 5–20% total time.
Significant speedup requires hitting the dispatcher's 30–40 ns + Tick
10–20 ns.

### 2.3 Implication for Phase 7 ordering

| Strategy | Cost section affected | Expected ROI |
|---|---|---|
| **A. Block-level JIT** | dispatcher overhead amortized | **High** — directly hits the bottleneck |
| **B.b/c GEP CSE / micro-op fusion** | function body micro-shrink | Low — body already small |
| **B.d nuw/nsw hint** | function body | Low |
| **B.e Tick / hook inlining** | per-instr fixed overhead | **Mid** — directly hits IRQ check |
| **C. Lazy flag** | flag write cost | **Mid** — depends on how many flag ops in spec |
| **D. Tier compilation** | compile time / hot block recompile | Only meaningful after block-JIT |
| **E. Mem-bus fast path** | extern call cost | **Mid–high** — saves trampoline per mem op |
| **F. Direct dispatcher table** | dispatcher overhead | **Mid** — overlaps partially with A |
| **G. Native AOT** | host runtime overhead | Mid — big effect on GB legacy ~300ms run |

Order: **A (block-JIT) → E (mem-bus fast path) → B.e (Tick inlining)
→ C (lazy flag) → F (dispatcher table) → G (AOT) → B.b/c/d (IR-level
micro-tweaks)**. Do A first because it amortizes dispatcher overhead in
one shot; most other optimisations become more salient post-block-JIT.

## 3. Why we kept O3 anyway

Despite no current gain:

- **No regression**: all 4 ROMs within noise
- **Future leverage**: once block-level JIT lands, functions get bigger
  (multiple instructions concatenated), O3 immediately matters — no need
  to flip back at that time
- **Compile time**: measured spec → IR → MCJIT startup cost rose from
  ~200 ms (O0) to ~250–300 ms (O3), acceptable
- **Unit tests + Blargg all green**: 345/345 unit + 9/9 Blargg cpu_instrs
  subtests pass on json-llvm path
