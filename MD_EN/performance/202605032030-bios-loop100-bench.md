# BIOS LLE loop100 perf — block-JIT is **28-31% slower** than per-instr (structural, not debug clutter)

> **Trigger**: User asked to bench 100LOOP + real BIOS, ARM/Thumb x
> per-instr/bjit 4-combo perf. Incidentally cleaned up dev-leftover
> hardcoded debug.
>
> **Finding**: When running loop100 under BIOS LLE, **block-JIT is slower
> than per-instr** (-28% to -31%). When running the same ROM under HLE,
> bjit is faster than pi by +27% to +40%. Switching boot mode flips
> bjit's advantage.
>
> **Cause**: bjit's cache-miss cost (build LLVM module → 5-pass
> optimisation pipeline → JIT add) cannot be amortised under the BIOS
> path's "lots of cold code" flow (vs HLE jumping straight into the cart
> entry's hot loop).
>
> **Decision**: Record as known characteristic, no fix. Followup perf work
> spun off separately (tier-block-compile-cost / cold-block-skip etc.).

---

## 1. Result

### 1.1 Bench (3-run avg, MIPS)

| combo | run1 | run2 | run3 | **avg** |
|---|---|---|---|---|
| arm   pi   (BIOS LLE) | 7.62 | 7.13 | 7.39 | **7.38** |
| arm   bjit (BIOS LLE) | 5.37 | 5.05 | 5.39 | **5.27** |
| thumb pi   (BIOS LLE) | 7.24 | 7.23 | 7.35 | **7.27** |
| thumb bjit (BIOS LLE) | 5.65 | 5.50 | 5.57 | **5.57** |

Discreteness (max-min / avg):
- arm pi: 6.6% (slightly high, possibly cache miss variation)
- arm bjit: 6.5%
- thumb pi: 1.7%
- thumb bjit: 2.7%

### 1.2 Comparison with HLE

| combo | HLE avg | BIOS avg | Δ |
|---|---|---|---|
| arm   pi   | 8.12 | 7.38 | **-9.1%** |
| arm   bjit | 10.31 | 5.27 | **-48.9%** |
| thumb pi   | 8.16 | 7.27 | **-10.9%** |
| thumb bjit | 11.43 | 5.57 | **-51.3%** |

**Per-instr running BIOS costs ~10% extra** — BIOS startup sequence overhead, as expected.

**Block-JIT running BIOS costs ~50% extra** — this is the structural cost of bjit mode.

### 1.3 Same-ROM bjit vs pi comparison

| boot mode | pi MIPS | bjit MIPS | bjit relative |
|---|---|---|---|
| HLE  arm   | 8.12 | 10.31 | **+27%** ← bjit wins |
| HLE  thumb | 8.16 | 11.43 | **+40%** ← bjit wins |
| BIOS arm   | 7.38 | 5.27 | **-29%** ← bjit loses |
| BIOS thumb | 7.27 | 5.57 | **-23%** ← bjit loses |

After switching to BIOS LLE, bjit's advantage disappears, and instead becomes a burden.

## 2. Root cause hypothesis

### 2.1 Cache miss → AddModule cost

block-JIT design:
1. First hit on a block start PC → `CompileBlockAtPc()`
2. Walk `BlockDetector.Detect()` to fetch N instructions from bus
3. Build LLVM module → `BlockFunctionBuilder.Build()` emit IR
4. **`HostRuntime.AddModule()`: set datalayout → 5-pass optimisation
   pipeline (mem2reg / instcombine / gvn / dse / simplifycfg) → add to JIT**
5. Cache fnPtr, next time call directly

Step 4 is the "fixed cost per cold block" on the hot path. Phase 7 H.a
brought the pipeline to 5 passes (was 4), increasing this cost.

### 2.2 Difference between BIOS and HLE

- **HLE path**: `apr-gba` skips BIOS, sets up cart entry @ 0x08000000
  directly. The loop100 ROM main loop is small (a few dozen instructions);
  after the cache warms up, N=million dispatches all directly hit cache,
  the optimisation pipeline amortises to infinity → bjit wins pi by 27-40%.

- **BIOS LLE path**: First runs through BIOS startup (CPU mode switching,
  setup IRQ vectors, IRQ handler entering/leaving SVC mode, SoftReset/IntrWait/SWI
  handler entry/exit paths, etc.). These paths are mostly "run-once cold
  code". Each cold path needs to be compiled once, with high cost; hit
  count is low → cannot amortise. Eventually **block compile cost > block
  execution cost**.

To validate the hypothesis (not done, leaving as followup):
- Add `BlockCompileMs` accumulator to `CompileBlockAtPc` + print at end
- Compare HLE/BIOS path's "total compile ms / total exec ms"

### 2.3 Why is per-instr BIOS only 10% slower?

per-instr instruction dispatch path:
- Pure lookup of `_fnPtrByDef` cache → directly call function pointer
- All spec emitter functions are pre-compiled once in `Compile()` (one
  big module, amortising 5-pass once), zero compile cost at runtime
- BIOS startup just runs more instructions, instruction-level does not
  trigger new compilation

So BIOS overhead = pure instruction count + IRQ overhead (ARM exception
entry/exit), about 10%, as expected.

## 5. Followup suggestions (do not block merge)

1. **Block compile cost amortisation**: track hot-block stats; blocks with
   hit-count < threshold skip the full 5-pass pipeline (only mem2reg +
   simplifycfg), promote after warming up. May pull BIOS bjit from 5.3
   back to 7-8 close to pi.
2. **Inline BIOS HLE replacement for SWI handlers**: most GBA games use
   BIOS mainly for SWI 1-FE. If the emulator has HLE intercepts for SWI
   and skips BIOS handler, bjit won't hit this cliff in game scenarios.
3. **Block detector boundary tuning**: `maxInstructions: 64` may be too
   large. What is the average N for cold blocks on the BIOS path? Smaller
   N → higher fixed compile cost ratio → more reason to skip compilation.

These are perf tuning, not correctness issues, kept for separate later commits.
