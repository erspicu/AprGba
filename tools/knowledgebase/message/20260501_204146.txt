Here are the answers to your 4 questions in order:

**1. Did setting CodeModel actually take effect in LLVM 20, or is it silently ignored?**
It is almost certainly being silently ignored or failing during relocation resolution. LLVM’s x64 Windows (COFF) backend historically has notoriously poor support for the Large code model. Even when explicitly requested, the backend will often fall back to emitting Small code model instructions (RIP-relative addressing) or the MCJIT dynamic linker will fail to process 64-bit absolute relocations for data symbols, truncating them to 32-bit offsets. 

**2. Could the issue be that the indirect call is unaffected by code model, and the crash site is the LOAD of the global?**
Yes, exactly. The x64 instruction for an indirect call via a register (`call rax`) natively supports full 64-bit absolute addresses. The distance between the JITted code and the trampoline does not matter at all. 

The crash is happening on the `load ptr`. Because the JIT is actually using the Small code model, it emits a RIP-relative instruction (`mov rax, qword ptr [rip + offset]`) to load the value *out of* `@memory_read_32`. Because your `NativeMemory.AllocZeroed` heap slot is more than 2GB away from where the JIT placed the code, the RIP-relative offset calculation overflows/truncates. The instruction reads from a garbage memory address, causing the `0xC0000005` Access Violation.

**3. Is there a way to force a specific virtual-address range for AddGlobalMapping's target so it's reachable by Small code model?**
Not easily from the managed C# side using standard OS allocators. `AddGlobalMapping` simply registers a pointer; it doesn't control memory placement. Because you don't know exactly where the JIT will allocate its memory until *after* compilation, you have a chicken-and-egg problem. 

To guarantee proximity, you would have to bypass `NativeMemory.Alloc` and implement a custom `LLVMMCJITMemoryManager` (via `LLVMCreateSimpleMCJITMemoryManager` in the C API) so you control the allocation of both the JIT's code pages and your data pages, ensuring they share the same 2GB block. Alternatively, you'd have to wait until JIT emission, find the code address, and use Windows `VirtualAlloc` to brute-force search for available memory pages near the JIT code.

**4. Or is the cleanest fix really to bake the trampoline address as an inttoptr constant via the global's INITIALIZER?**
Yes, this is by far the cleanest and most reliable fix for this architecture limitation. 

By setting the global variable's initializer via `LLVMSetInitializer` to your constant `inttoptr` value, you stop treating it as an "external" mapped symbol. Instead, LLVM will allocate this global variable directly inside the JIT's own read-only data section (`.rdata`). The JIT memory manager strictly guarantees that a module's code (`.text`) and data (`.rdata`) are allocated adjacent to each other. 

Therefore, the RIP-relative `load` instruction will successfully find the global, load the full 64-bit absolute trampoline address you baked into it, and the subsequent register-based `call` will execute perfectly.
