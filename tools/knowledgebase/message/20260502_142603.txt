Based on your detailed state dump, you have run into a **CPU emulation bug or memory bus issue, not the expected logo verification loop.** 

Here is the technical breakdown answering your questions:

### 1. Is this the logo verification infinite loop?
**No.** The standard BIOS behavior for a bad Nintendo logo is to lock up *after* the boot intro animation finishes. 
* Because `HALTCNT` was never written, the BIOS never called `SWI 0x05` (VBlankIntrWait). This means the 2-3 second intro animation **never played**. 
* Look closely at the disassembly at `0x19BA`:
  `MOV R0, #6`
  `ADD R0, R0, R2`
  `LSL R0, R2, #2`  *(Note: this is `R0 = R2 << 2`)*
  This calculates `6 + R2` into `R0` and immediately overwrites it with `R2 << 2`. Compilers do not emit dead code like this. This strongly implies the CPU has derailed and is **executing data** (like compressed graphics or audio) as instructions. 
* Furthermore, `R4 = 0x06C07098` points way outside the 96KB VRAM (`0x06000000`–`0x06017FFF`), meaning a pointer math loop ran wildly out of bounds.

### 2. What is the standard emulator workaround for logo checks?
Even though it's not the cause of *this specific hang*, real BIOS LLE will eventually refuse to jump to `0x08000000` without a valid logo. 
* **The standard workaround:** Immediately after loading the cart ROM into your emulated memory buffer, check if the logo is missing. If so, hard-patch `0x080000A4` to `0x080000DF` (156 bytes) with the correct, canonical Nintendo logo byte array. (Optionally, correct the header checksum byte at `0x080000BD`).

### 3. Does BIOS verify the logo BEFORE waiting for VBlank?
**Yes and No.** The BIOS reads and calculates the checksum of the cart logo *before* the animation to decide whether to drop the "Nintendo" text or just show "Game Boy". However, it **does not** enter its infinite failure loop at this stage. It will still use VBlank interrupts (`SWI 0x05`) to play the base "Game Boy" fade-in animation regardless of cart validity. Since you haven't seen a single `HALTCNT` write, it crashed before or during the initial memory clearing/decompression phase.

### 4. Are there known CPU/Bus bugs that pass js_molka but crash the BIOS?
Yes. Instruction-level tests like js_molka are fantastic but often miss architectural pipeline quirks and memory-bus edge cases that the BIOS boot sequence specifically hits:

* **Thumb PC-Relative Loads (`LDR Rd, [PC, #imm]`):** In Thumb mode, `PC` reads as `InstructionAddress + 4`, but for PC-relative loads, the base address **must be forced to be word-aligned** (`(PC & ~3)`). If you don't align the PC before adding the offset, the BIOS reads the wrong literal pool data, corrupts a branch target, and jumps into a data block (which perfectly matches your symptoms).
* **Open-Bus Reads:** The BIOS intentionally reads from unused memory areas. If your bus returns `0x00` or `0xFF` instead of true GBA open-bus behavior (which usually returns the last fetched instruction from the prefetch buffer), the BIOS can misinterpret it as an infinite loop counter or a bad size header.
* **Empty `LDM/STM` Lists:** An `LDM` with an empty register list (`0x0000`) in ARMv4 actually loads `PC`. JS_Molka doesn't perfectly test all weird `LDM/STM` edge cases (like base-register writeback when the base register is also in the list, or empty lists).
* **BIOS Read Protection:** The real BIOS protects itself. If the CPU PC is `> 0x00003FFF`, data reads (`LDR`) to the BIOS region (`0x00000000` - `0x00003FFF`) must return open bus, not the BIOS data. If you implemented this loosely, ensure you aren't accidentally blocking the BIOS from reading its *own* data while executing from inside itself.

**Triage Recommendation:** Put a breakpoint on branches (`BX`, `POP {PC}`) right before `PC` hits `0x1AC3` (where `R14` indicates it came from). You likely have a corrupted literal load or a loop counter that didn't terminate properly during the `0x06000000` clearing phase.
