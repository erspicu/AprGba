Based on the detailed trace and the symptoms you described, the most probable class of bug lies in **Thumb PC-relative branch target calculation**, specifically an edge case involving conditional branches with an **offset of `+0`**. 

You left a crucial clue in your disassembly notes that likely reveals the exact bug in your JIT/emulator:
> `0x1B0C: D400 BMI +0 (effectively no-op?)`

**`BMI +0` is NOT a no-op.** This single misunderstanding is almost certainly the root cause of the infinite loop.

### The Architectural Rule
In Thumb mode, the PC used for branch calculations is always **Instruction Address + 4** (to account for the pipeline), not `+2`. 

When the CPU evaluates `0x1B0C: D400 BMI +0`:
1. The base `PC` for the calculation is `0x1B0C + 4 = 0x1B10`.
2. The offset is `+0` (0 halfwords = 0 bytes).
3. The branch target is exactly **`0x1B10`**.

Notice what is at `0x1B0E` (the instruction immediately following the `BMI`):
`0x1B0E: E753 B -346 -> 0x19B8`

A `B<cond> +0` instruction is a standard ARM compiler idiom used to conditionally **skip the next 16-bit instruction**. 
If `R5` is negative (N=1), the `BMI` should branch to `0x1B10`, perfectly jumping *over* the unconditional backward branch at `0x1B0E` and successfully exiting the loop!

### The Likely Bug in Your Emulator
Because you annotated it as `(effectively no-op?)`, it is highly probable that your LLVM translation layer explicitly optimizes out branches with an offset of `0` (e.g., `if (offset == 0) return; /* skip emitting jump */`), or your JIT incorrectly calculates the Thumb `PC` base as `Address + 2` in some branch evaluation contexts.

If your emulator treats `BMI +0` as a no-op, execution will blindly fall through to `0x1B0E` regardless of the condition flags. 

**This perfectly explains all of your symptoms:**
1. **The Infinite Loop:** The exit condition is evaluated correctly (`SUB R5, #1` does set N=1 when R5 is negative), but because the emulator treats the branch as a no-op, it falls through to `B -346`. The loop loops forever.
2. **R5 going deeply negative:** R5 decremented past `-1` and just kept going to `-1,331,207` because the `BMI` branch that was supposed to break it out was effectively ignored.
3. **R14 is STABLE at `0x1AC3`:** The `BHI +0x18` forward branch at `0x1AEE` is properly skipping the `BL` instructions inside the loop, preserving R14. 
4. **jsmolka passes:** Standard test ROMs (like jsmolka) test conditional branching via jumps to explicit `pass`/`fail` labels (non-zero offsets). They rarely test the compiler-specific `+0` instruction-skip idiom, allowing a `0`-offset optimization or off-by-2 bug to slip through undetected.

### What is the function doing?
You are right to suspect decompression. This function is likely **SWI 0x11 (LZ77UnCompReadNormalWrite8bit)** or **SWI 0x12 (16-bit)**. 
During the GBA boot sequence, the BIOS decompresses the animated Nintendo logo using a tight LZ77 loop. `R5` is the remaining uncompressed size counter. It counts down to 0, hits `-1`, triggers the `BMI` flag, and is supposed to skip the loop back-edge to return. Because it couldn't skip the back-edge, the BIOS never finishes drawing the logo, never reaches the `HALT` (SWI 02h), and never boots the cart.

**Next Step to Fix:**
Check your LLVM IR generation for `B<cond>` (Thumb format 16). Ensure `offset == 0` is NOT being discarded, and explicitly verify that your branch target calculation strictly uses `Target = (InstructionAddr + 4) + (SignedImm8 * 2)`.
