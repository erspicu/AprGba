Here is a detailed analysis of your emulator's divergence and direct answers to your architectural questions. 

The symptom you are seeing—`POP {R3}` reading a `0` because an earlier `PUSH {..., LR}` wrote a `0`—is a classic block-JIT bug. R14 (LR) was either incorrectly zeroed out, pulled from an uninitialized register bank, or loaded with a stale block-start PC.

### Most Likely Culprits for LR == 0

**1. The Bank-Swap Bug (Your Hypothesis 5)**
If your `MSR CPSR` (or Exception entry) bank-swap logic captures `oldMode` at **JIT-compile time**, this is almost certainly your bug. The GBA BIOS switches modes dynamically (e.g., jumping from SYS to IRQ to SVC). If a block executes twice under different starting modes, or if the mode is modified by a register whose value is unknown until runtime, a JIT-time baked `oldMode` will swap the wrong register banks. 
*   **Result**: R14 is read from an uninitialized bank (yielding `0`), which then gets pushed to the stack.
*   **Fix**: Your host extern call should only take `newMode` (or `newCpsr`) and the `State*`. Inside the extern host function, read `oldMode = state->Cpsr & 0x1F` dynamically. 

**2. Stale PC Read into LR (Hypothesis 1 & 2)**
In Strategy 2, `GPR[15]` in memory contains the stale `block_start_pc`. If an instruction computes `LR = PC` (like a manual indirect call setup or `BL`) and it fails your static `read_reg` check, it will fallback to a dynamic IR load from `GPR[15]`.
*   If your emulator uses LLVM IR instructions to extract register fields (e.g., `%rm = and %instr, 15`), LLVM's `Value` is an `Instruction`, not a `ConstantInt`. Your C# check `if (val is ConstantInt c && c == 15)` will fail! It will emit a dynamic load from memory, fetching the stale `block_start_pc`. If the block started near `0x00000000` (BIOS start), LR becomes `0`.

**3. Direct Memory Store Bypassing `PcWritten` (Hypothesis 3)**
If an ARM `LDM {..., PC}` or Thumb `POP {..., PC}` emitter writes to `GPR[15]` using a direct `GEP` + `Store` IR sequence, it bypasses the `MarkPcWritten` flag that your standard `write_reg` wrapper sets. 
*   **Result**: The block doesn't terminate. Execution falls through to the next instruction sequentially. At the end of the block, the executor sees `PcWritten == 0` and overwrites `GPR[15]` with the linear advance, completely erasing the branch.

### Answers to Your Specific Questions

#### A. Standard pitfalls for reading dynamically-indexed registers?
In ARM7TDMI, **register indices are never truly dynamic**—they are always fully encoded in the instruction word. Because you have a constant `InstructionWord` per block, you should **never use LLVM IR to extract register fields**. 
*   **Pitfall**: Emitting LLVM IR to mask/shift the instruction word, resulting in an LLVM `Value` representing the index. Because you have disabled optimization passes, LLVM will not constant-fold this.
*   **Solution**: Decode the register indices entirely in C# during JIT compilation (`int rd = (instr >> 12) & 0xF;`). Then pass the C# `int` to `read_reg(int index)`.

#### B. Canonical way to handle reads of R15 when the index isn't a JIT-time constant?
As established above, it *should* be a JIT-time constant. However, if you absolutely must emit a dynamic read (e.g., for unpredictable Coprocessor register transfers), the canonical mGBA/Dynarmic way is to emit an IR `select` or `br`:
```llvm
%is_r15 = icmp eq i32 %dynamic_index, 15
%pipeline_pc = add i32 %current_instr_pc, 8 ; (Baked as constant)
%mem_val = load i32, ptr %dynamic_gep
%result = select i1 %is_r15, i32 %pipeline_pc, i32 %mem_val
```

#### C. LDM/STM rlist: Will LLVM DCE false-condition branches without passes?
**No.** If `instcombine` and `mem2reg` are disabled, LLVM's FastISel / SelectionDAG will directly translate your IR. If you emit 16 `if (cond)` blocks in IR, it will emit 16 compare-and-branch sequences in machine code.
*   **The Clean Fix**: Because `InstructionWord` is constant, evaluate the `rlist` in C# *before* emitting IR.
```csharp
uint rlist = InstructionWord & 0xFFFF;
for (int i = 0; i < 16; i++) {
    if ((rlist & (1 << i)) != 0) { // C# evaluation!
        // Emit IR strictly for registers that are actually transferred.
    }
}
```
This entirely eliminates the dead code, avoids FastISel bloat, and prevents any speculative reads.

#### D. Cleanest way to verify "did block fn write PC mid-block but fail to set PcWritten"?
Inject an assertion snippet directly into your JIT builder's per-instruction `post` step, right before you check `PcWritten`. This will abort the emulator the instant a flag is missed:
```llvm
; In the post[K] block:
%actual_pc = load i32, ptr %Gpr15_Address
; %expected_pc is whatever the PC was at block_start (stale PC)
%pc_changed = icmp ne i32 %actual_pc, %expected_pc 

%flag = load i1, ptr %PcWritten_Address
%missed_flag = and i1 %pc_changed, (not %flag)

br i1 %missed_flag, %trap_miss, %continue_post

trap_miss:
; Call a host extern to throw a fatal C# exception with the instruction's PC
call void @Host_TrapMissedPcWritten(i32 %current_instr_pc)
unreachable
```

#### E. Documented LLVM bugs with many small basic blocks interacting with FastISel?
Yes. FastISel is extremely naive and struggles heavily with register allocation across many small basic blocks (especially 100+ BBs with complex control flow). It can run out of registers, resulting in heavy stack spilling, or simply fail and fallback to SelectionDAG. 

Furthermore, you noted that `instcombine` miscompiled your BIOS LLE. **`instcombine` does not miscompile valid IR.** If `instcombine` breaks your emulator, it guarantees your IR contains Undefined Behavior (UB). FastISel happens to generate machine code that masks the UB by coincidence.
*   *Common UB in emulator IR*: ARM `LSL`/`LSR`/`ASR` instructions can shift by values `>= 32`. LLVM's `shl` and `lshr` instructions have UB if the shift amount is `>= bitwidth`. You must clamp or cleanly mask shift amounts in IR before applying them, otherwise `instcombine` turns the entire block into `poison` or `undef`.

#### F. Debug techniques for finding the exact miscompiling IR instruction?
Do not try to guess by looking at the endpoint crash; find the exact cycle of divergence using an **Instruction Execution Trace Diff**.
1.  Inject an extern call at the `pre` block of every instruction in your JIT: `LogState(PC, R0...R14, CPSR)`.
2.  Do the exact same in your per-instruction executor.
3.  Run both for 1 frame, pipe to `block.log` and `per_instr.log`.
4.  Use a diff tool (Beyond Compare, VSCode, KDiff3). Because the divergence is within 280k cycles, the logs will easily fit in memory.

Scroll to the exact line where the diff shows `R14` (or any other state) changing in the block-JIT but not in the per-instr run. You will instantly know *exactly* which ARM/Thumb mnemonic caused the corruption, and you can dump the LLVM IR specifically for that single block to spot the error.
