Here is the expert insight into your divergence, addressing your questions directly.

**The Smoking Gun (Question 3: IRQs)**
Your finding that **BJIT delivered 0 IRQs while PI delivered 263** is the absolute root cause of this crash. 

During the GBA/NDS BIOS initialization, the system uses `SWI 4` (IntrWait) or `SWI 5` (VBlankIntrWait) to halt the CPU until hardware sets specific flags in the BIOS data area (around `0x03007FF0`). 
In your PI emulator, IRQs fire, the BIOS IRQ handler (which pushes context to the IRQ/SVC stack) executes, acknowledges the VBlank, updates the BIOS wait-flags, and gracefully resumes. 
In your BJIT emulator, because IRQs are never delivered, the BIOS spin-loop eventually times out, or a fallback code path executes. The path jumping to `0xB56` and popping `0` from `0x03007E9C` is a **junk execution path** caused by the CPU missing the expected state change that the IRQ handler was supposed to perform. Function `0x874` correctly zeroed that memory block, but because the IRQ never fired, the expected return address was never written there. PI avoids `BX 0` because the IRQ *does* fire, altering the control flow before or at this critical junction.

*Why do JITs miss IRQs?* Block-JITs typically only check for pending interrupts at block boundaries. If your emulator relies on a tight block loop (like the memory clear loop at `0x728`) and your block compiler doesn't embed cycle-counting and IRQ-polling *inside* backward branches or block exits, the CPU can get trapped in a block, missing edge-triggered IRQs or delaying them until the BIOS state machine breaks.

**Question 1: BIOS Code Pattern**
Jumping "into the middle" of functions is normal in the GBA BIOS. The BIOS heavily utilizes ARM/Thumb interworking and shared epilogues to save ROM space. Context switches, SWI unpacking, and IRQ returns frequently manipulate the stack and `LR` to return to shared cleanup routines. The sequence you see is the JIT dutifully executing a corrupted state machine (due to the missed IRQ), resulting in an underflow or misalignment of the expected call stack.

**Question 2: LLVM Optimization Pipeline**
Disabling LLVM optimizations (running at `-O0`) does **not** cause unsafe memory reordering. In fact, `-O0` is pedantically sequential: every LLVM IR load/store results in a direct memory access in the exact order written. If this were an LLVM reordering issue, it would happen *with* optimizations enabled. The stack timing corruption is purely logical, stemming from the missing IRQ handler execution.

**Question 4: Instrumentation Strategy**
To find the exact divergence point:
1. **Log IRQ Checks:** The divergence happens exactly where PI acknowledges an IRQ and BJIT ignores it. Log every time the `CPSR.I` bit is unmasked AND the hardware IRQ line is high. 
2. **State Hashing:** Instead of comparing every instruction, calculate a quick hash (e.g., FNV-1a) of `R0-R15` + `CPSR` at the end of every JIT block. Print `[Instruction Count] [PC] [State Hash]`.
3. **Compare the Logs:** Run this in both PI (hashing only at block-equivalent boundaries) and BJIT. The logs will match perfectly until the specific block where PI branches to the IRQ vector (`PC=0x18`) and BJIT falls through.

**Fix Action:** Ensure your BJIT compiles an "IRQ pending" check at the end of every block, and crucially, **before backward branches** (to prevent infinite looping without checking hardware state).
