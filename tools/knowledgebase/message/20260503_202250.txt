Transitioning a block-JIT from a fixed-width RISC architecture (ARM) to a variable-width CISC-lite architecture (LR35902/Z80) requires a fundamental shift in how the frontend parses instructions. The challenges you are facing are classic milestones in dynamic recompiler (dynarec) development. 

Here is how industry-standard emulators (QEMU, Dolphin, mGBA, Dynarmic) handle these exact architectural hurdles, tailored for your LLVMSharp/.NET framework.

---

### 1. Variable-Width Instruction Handling & Block Detection

**The Industry Standard: Synchronous Pre-Decode**
Do not use lazy decoding or defer width calculation to the execution phase. In standard block-JITs, the frontend translation phase (detection) must pre-decode the entire block sequentially. 

In **QEMU’s TCG (Tiny Code Generator)**, the block translation loop (`translator_loop` in `accel/tcg/translator.c`) fetches memory, decodes an instruction, emits TCG opcodes (your LLVM IR equivalent), and dynamically advances the virtual PC by the exact length of the decoded instruction. It repeats this until a terminal condition is met (branch, page boundary, block size limit). 

**How to adapt your JSON-driven framework:**
Your `BlockDetector` should transition from a simple array-slicing approach to a streaming cursor. 
1. Read the opcode at `PC`.
2. Look up the instruction spec in your JSON model.
3. Advance `PC` by `instruction_length`.
4. Append the decoded instruction metadata to an intermediate block list.
5. Loop until terminal instruction.

Because your emitter already uses Strategy 2 (baking in PC offsets), pre-decoding the block gives you the exact byte-offset of every instruction. When you walk the block to emit LLVM IR, you simply add the accumulated byte-offset to the block's base PC.

**Explicit vs. Inferred Length in JSON Spec:**
You should **explicitly define the byte length** in your JSON spec (e.g., `length: 2`), rather than inferring it from operand types. While inference (`imm8` = +1) works for the Game Boy, it will completely collapse if you ever target x86, where instruction length is a complex matrix of prefixes, ModR/M bytes, and SIB bytes. By putting `length: <int>` directly in the JSON, your `BlockDetector` remains architecture-agnostic. 

---

### 2. Narrow-Int Register Allocation in LLVM (`i8` vs `i32`)

**The Verdict: Promote to `i32` internally.**

While LR35902 has 8-bit registers, emitting pure `i8` arithmetic in LLVM IR targeting x86-64 introduces two severe bottlenecks for emulator performance:

**A. The x86 Partial Register Stall / False Dependency Issue**
x86-64 has aliased registers (`AL` is the lower 8 bits of `RAX`). If LLVM emits an 8-bit add (`add al, bl`), modern x86 processors must ensure the upper 56 bits of `RAX` remain unchanged. Historically, this caused partial-register stalls (P6 architecture). Modern CPUs handle this via register renaming, but doing chains of 8-bit operations can still generate false dependencies on the upper bits, preventing Out-of-Order (OoO) execution from reordering your JITted instructions efficiently. 

**B. The Half-Carry (H) Flag Problem**
The Game Boy's DAA (Decimal Adjust Accumulator) instruction relies entirely on the Half-Carry flag (bit 4 carry). If you use LLVM's `add i8 %a, %b`, the carry-out from bit 3 to bit 4 is lost to LLVM. You would have to use LLVM intrinsics (`@llvm.uadd.with.overflow`) which are cumbersome and don't natively capture half-carries.

**The Solution (used by Dolphin and Yuzu):**
At the start of your block, load your 8-bit GPRs from your state struct and immediately zero-extend (`zext`) them to `i32`. Do all your math in `i32`.
By doing this, calculating the notoriously difficult Z80 flags becomes a trivial bitwise operation that LLVM optimizes perfectly:

```c
// Emitting a Game Boy ADD A, B in i32 IR:
i32 result = a + b;
i32 half_carry = (a ^ b ^ result) & 0x10; // InstCombine loves this
i32 carry = (a ^ b ^ result) & 0x100;
// update flags...
```
At the end of the block (or at a store-to-memory), truncate (`trunc`) the `i32` back to `i8`. LLVM's `InstCombine` and `DAGCombiner` passes are incredibly aggressive. If it sees `i8 -> zext -> math -> trunc -> i8`, it will often compile it down to highly optimized native x86-64 instructions using 32-bit registers (`add eax, ebx`), masking out the garbage natively. This eliminates partial register stalls and gives you easy access to flag bits.

---

### 3. Prefix Byte Handling for Block Boundaries

**The Issue:** Modeling `0xCB` as an ISA switch (`switches_instruction_set: true`) is semantically incorrect for block JITs. `0xCB` is not a state switch (like ARM to Thumb); it is simply the first byte of a multi-byte instruction. Breaking the block here ruins instruction cache locality and LLVM optimization scope.

**The Industry Standard: Decoding Tries (Prefix Trees)**
To make your JIT robust enough to handle Game Boy, Z80 (`DD`, `FD`, `ED`, `CB`), and x86 (REX, VEX), you must decouple *memory bytes fetched* from *logical instructions emitted*.

Instead of a flat array mapping 256 opcodes to JSON specs, your JSON parser should construct an **Opcode Trie** in memory during framework initialization. 

In your JSON spec, remove `switches_instruction_set: true` from the `0xCB` entry. Instead, define it as an instruction with a `sub_decoder` or `prefix_table` property.

When your `BlockDetector` is walking the memory stream:
1. It reads byte `0xCB`.
2. It looks up `0xCB` in the root Trie node. The node says: "I am a prefix, advance PC by 1 and read the next byte from Table CB."
3. It advances `PC`, reads `0x37` (SWAP A).
4. Table CB maps `0x37` to the actual JSON spec for `SWAP A`.
5. The `BlockDetector` emits **one logical instruction** (`SWAP A`) to the block IR, with a total parsed byte length of 2.

**Real-world reference:** 
Look at how **QEMU** handles x86 prefixes in `target/i386/tcg/translate.c`. It maintains a `DisasContext` struct. When the fetch loop encounters a prefix byte (e.g., `REP` or `Lock`), it doesn't emit an instruction. It sets a flag in `DisasContext` (`s->prefix |= PREFIX_REP`) and loops back to fetch the next byte. Once it hits a terminal opcode, it uses the accumulated prefix state to select the correct code generation path.

For your framework, this translates to:
* Treat an instruction not as "1 byte = 1 instruction", but as a path through a state machine. 
* The `BlockDetector` reads bytes and traverses the JSON-defined Trie until it hits a terminal "leaf" node (the actual instruction). 
* It then hands that *single leaf node* (which may represent 1, 2, or 4 bytes of fetched memory) to the LLVM IR emitter. 

By modeling prefixes as Trie traversal rather than ISA state switches, your block-JIT will happily swallow long chains of `CB` instructions within a single, highly optimized LLVM basic block, drastically improving your MIPS.
