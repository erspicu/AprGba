Context: I'm building a JSON-driven CPU emulator framework in C# (.NET 10) using LLVMSharp.Interop 20.x with ORC LLJIT. CPU specs are JSON files describing instruction encoding + micro-op step sequences; the framework compiles spec → LLVM IR → native code per CPU instruction (per-instr dispatch) and per basic block (block-JIT) for hot paths.

Already shipped: ARM7TDMI (ARM 32-bit + Thumb 16-bit) + Sharp LR35902 (Game Boy DMG). Block-JIT works for both. Per-instr block-JIT lockstep diff harness shows zero divergence on Blargg cpu_instrs/01-special (CPU correctness tests).

Now hitting a class of bugs I'd like to handle generically. The concrete instance is LR35902's EI instruction quirk: EI sets IME=1 only AFTER the next instruction completes. Per-instr backend handles this with a host-side `_eiDelay` counter that the outer loop decrements per StepOne; block-JIT runs N instructions per StepBlock so the per-instruction decrement granularity is lost — IME flips after the WHOLE block instead of after just one more instruction. Tests like Blargg cpu_instrs/02-interrupts subtest "EI" depend on the precise per-instruction granularity.

Similar "delayed-effect" patterns exist in other CPUs:
- Z80 EI (same 1-instr delay as LR35902 — LR35902 inherits from Z80)
- ARM7TDMI: I'm not aware of any built-in delayed flags but ARM cores in general have things like CPSR mode-switch causing pipeline reload that takes effect a fixed number of instructions later
- 6502: NMI delivery is sampled before instruction fetch but raises only after current instruction completes
- 65816: `XCE` / `SEP`/`REP` mode-switch instructions whose effect on register width takes the next instruction
- RISC-V: `fence.i` invalidates instruction-cache, takes effect at next fetch
- x86: `STI` enables interrupts but they remain blocked through the NEXT instruction (Z80-like)
- M68K: `STOP` has subtle wakeup ordering
- SH-2: delayed branches (this isn't quite "delayed effect" but illustrates the design principle of "next instruction modifies behavior")

I want to express these cleanly in the JSON CPU spec rather than hand-coding LR35902-specific logic in my BlockDetector. The properties I care about:

(1) **Spec syntax should describe the SEMANTIC** ("this op causes state change S to take effect after N more instructions execute"), not the implementation. Reasoning: describing the implementation defeats the purpose of a JSON-driven framework. Same spec must work for both my per-instr backend AND my block-JIT backend.

(2) **Block-JIT cost should be low** (single-digit % perf hit ideally). Currently block-JIT is ~+70% over per-instr; would hate to give half that back. Acceptable: a small counter decrement per instruction inside a block; not acceptable: forcing 1-instruction blocks after every delayed-effect op (kills the win).

(3) **Should support multiple parallel pending effects**. EI delays IME-flip; if some other instruction simultaneously delays a different bit (hypothetical but possible in Z80 with multiple shadowed banks), the framework needs to handle both pending without conflict.

(4) **Generalizes to "branched delays"**: in some CPUs the delay isn't a fixed N; it's "until next branch is taken" or "until next memory load completes". I don't need this YET but the design shouldn't paint into a corner.

What's your recommendation? Specifically:

(A) **JSON spec syntax**: How would you express the EI quirk? Inline in the instruction's `steps` (a new micro-op like `defer_state_change`)? At the format level (a `post_completion_action`)? Top-level CPU declaration?

(B) **Block-JIT IR pattern**: Given a spec like in (A), what IR pattern makes this cheap? My intuition: maintain a small `pending_actions` bitmap in CPU state, with each bit = "this delayed action is pending"; each block instruction checks the bitmap before its own work and applies/clears the action; cost is one load + branch per instruction (LLVM's cmov should handle most cases). Is this what dynarec engines like Dolphin/Dynarmic/mGBA actually do? Or do they use something different (per-block "deferred-action queue", per-instruction phantom step injected by spec compiler, etc)?

(C) **Existing dynarec patterns**: How do real-world dynarec emulators handle this class of problem? Citations to source code or specific papers welcomed (I can read C++/Rust/Go). Specifically interested in how Dynarmic handles ARM CPSR mode-switch delays, how Dolphin handles GameCube interrupt timing, how Blargg's own Z80 emulator handles EI in its dynarec mode (if any).

(D) **Alternative architectures**: Is there a totally different way to model this that I'm missing? E.g., "delayed effects are a special case of micro-coded instructions where the spec compiler unrolls them into multiple synthetic instructions in the block"? Or "block-JIT detector ends block at any instruction with deferred semantics, then resumes with the deferred action consumed"?

The goal is for my framework to handle ANY future CPU's delayed-effect patterns by just describing them in JSON, without me touching C# code. ~800-1500 words please, with specific implementation suggestions I can prototype.

========================================================================
This is a classic problem in dynamic recompilation (dynarec). The core friction comes from the fact that your per-instruction backend operates strictly in the **dynamic domain** (evaluating state at runtime), while your block-JIT operates in the **static domain** (knowing the sequence of upcoming instructions at compile time). 

Your intuition about needing a generic solution is spot-on. Hand-coding these quirks leads to unmaintainable JIT detectors. Here is a comprehensive architecture to handle delayed effects generically, satisfying all your constraints.

---

### (A) JSON Spec Syntax: The `defer` Micro-Op

To keep the JSON semantic, you should treat delays as a temporal modifier applied to standard micro-ops. Instead of writing a new micro-op per quirk, introduce a generic `defer` control-flow block.

For the LR35902 `EI` instruction, the JSON would look like this:

```json
{
  "mnemonic": "EI",
  "encoding": "11111011",
  "steps": [
    {
      "op": "defer",
      "delay_type": "instruction_count",
      "delay_value": 1,
      "body": [
        { "op": "set_flag", "flag": "IME", "value": 1 }
      ]
    }
  ]
}
```

**Generalizing to your other examples:**
*   **x86 STI / Z80 EI:** Exactly the same as LR35902.
*   **Branched Delays (SH-2 / RISC-V `fence.i`):** Change the delay type. 
    ```json
    "delay_type": "until_condition",
    "condition": "branch_taken"
    ```
*   **Parallel pending effects:** The JSON doesn't need to care. If an `EI` is followed by another instruction with a `defer` block, the JIT framework will simply track both independently.

This syntax describes *what* happens and *when*, completely divorcing the semantics from how your backends lower it to C#/LLVM IR.

---

### (B) Block-JIT IR Pattern: Static Resolution + Dynamic Fallback

Your intuition—a dynamic `pending_actions` bitmap checked at every instruction—is how a per-instruction interpreter handles this. **Do not do this inside a block-JIT.** It throws away the primary advantage of block compilation: ahead-of-time knowledge.

Instead, use a **Compile-Time Pending Queue** combined with a **Dynamic Fallback**.

**1. Static Resolution (Zero runtime cost)**
When your block compiler loops over the instructions to generate LLVM IR, it should maintain a compile-time list of deferred actions: `List<PendingAction>`. 

1.  JIT compiles Instruction 1 (`EI`). It sees the `defer` block. Instead of emitting LLVM IR for `IME=1`, it pushes the AST of the `body` into the compile-time list: `Pending[Delay=1, AST=IME=1]`.
2.  JIT moves to Instruction 2 (`NOP`). 
3.  *Before* emitting the IR for Instruction 2, the JIT decrements the compile-time delay of all items in `Pending`. 
4.  The `EI` delay hits 0. The JIT pulls the `IME=1` AST out of the queue and generates LLVM IR for it *right here*, before the `NOP`.

**Result:** The generated machine code executes the `EI` logic, then the `IME=1` logic, then the `NOP` logic. There are zero dynamic checks, zero bitmaps, and zero performance loss. 

**2. Dynamic Fallback (Handling Block Boundaries)**
What if `EI` is the *last* instruction in the block, or is immediately followed by a branch that ends the block? The compile-time queue cannot resolve statically. 

*   **Block Epilogue:** When terminating a block, if the compile-time `Pending` queue is not empty, the JIT emits LLVM IR to serialize the pending actions into the CPU state (your proposed bitmap/counters).
*   **Block Preamble:** When entering *any* block, the JIT must emit a quick check: `if (cpu->pending_bitmap != 0) goto HandlePending;`. 

Because block linking usually ensures long runs of code, the block epilogue/preamble fallback will rarely trigger. You get the 1-instruction granularity for free within the block, and fallback to dynamic counters only across block boundaries.

---

### (C) Existing Dynarec Patterns

How do real-world emulators handle this? They almost universally prefer **Static Resolution** where possible, and **Events** for the rest.

**1. Delay Slots (MIPS / SH-2)**
Emulators like **Ares** or standard MIPS dynarecs handle branch delay slots strictly at compile-time. When the JIT encounters a branch, it does not emit the branch IR yet. It fetches the *next* instruction, emits its IR, and *then* emits the branch IR. This is the exact compile-time reordering concept described in Section B.

**2. State-Tracking and Lazy Evaluation (Dynarmic)**
**Dynarmic** (a popular ARM dynarec used in Yuzu/Citra) uses an intermediate representation pass that explicitly tracks CPU state (like CPSR flags) statically within a basic block. If a pipeline-flushing mode switch occurs, Dynarmic will usually force an end-of-block. Because mode switches are relatively rare in hot loops, forcing a block break costs almost nothing. 

**3. Cycle-Timed Event Queues (Dolphin / mGBA)**
For delayed effects measured in *cycles* (rather than instructions)—like GameCube interrupts or GBA DMA triggers—emulators use a global hardware event queue. 
*   They calculate exactly how many cycles a basic block will take.
*   The block preamble checks: `if (cycles_to_run >= next_hardware_event) { exit_block_early(); }`.
*   This is highly efficient for cycle-based delays, but overkill for LR35902 `EI`, which is strictly tied to the instruction fetch pipeline, not a physical hardware timer.

**4. x86 `STI` (DOSBox / Bochs)**
In many x86 emulators, `STI` sets a dynamic `interrupts_inhibited = true` flag that naturally expires. However, in optimized JITs, `STI` is usually treated as a block-terminating instruction. The JIT ends the block, emits the next single instruction in its own tiny block with interrupts masked, and then resumes normal execution.

---

### (D) Alternative Architecture: The "Phantom" Instruction Injection

If you want to keep your JIT strictly linear and avoid maintaining a compile-time queue inside the LLVM compiler pass, you can handle this via an **IR Pre-Pass (AST Transformation)**.

Before you hand a block of JSON-derived ASTs to LLVMSharp, run a cleaner pass over the sequence of instructions:

1.  The BlockDetector hands you a list of 10 instructions to compile.
2.  Your `SpecCompiler` iterates through them. 
3.  It finds Instruction 4 (`EI`) with a `defer(delay=1)` block.
4.  The compiler removes the `defer` wrapper from Instruction 4, extracts the `body` (`IME=1`), and physically injects it into the AST of Instruction 5 as a "phantom" micro-op at the very beginning of Instruction 5's steps.
5.  If Instruction 4 is the end of the block, it injects a "serialize_pending_state" micro-op instead.

You then pass this mutated AST sequence to the LLVM IR generator. 

**Why this is powerful:**
Your LLVM IR generator remains completely "dumb." It doesn't need to know about time, queues, or delays. It just blindly compiles the micro-ops it receives in order. The "magic" of delayed execution is entirely handled by a lightweight C# AST transformation pass prior to LLVM compilation. 

### Summary Recommendation for Your Framework

1.  **JSON:** Express the semantic using a `defer` wrapper micro-op with a `delay_type` (instruction_count, branch_taken, etc.).
2.  **Architecture:** Implement the **IR Pre-Pass (Phantom Instruction)** pattern (Section D). It keeps your LLVM generator clean.
3.  **Cross-Block:** For `defer` actions that bleed past the end of a basic block, have the Pre-Pass inject a `serialize_pending_state` micro-op that writes to a `pending_bitmap` in your CPU state. Add a fast-path check to your block preamble to resolve it dynamically.
4.  **Performance:** This guarantees that 95% of delayed effects are resolved at compile-time with **zero runtime overhead**, preserving your block-JIT performance gains, while safely handling edge cases across block boundaries.
